Big-O Notation

Categorizes algorithms on time/memory based on input.

Why?​

  • Helps us make decisions on what data structures and algorithms to use.

Fundamental concept​

As input grows, how fast does computation or memory grow?

Important concepts​

Growth is with respect to input​

fn sum_char_codes(input: String) -> u32 {
    let mut i = 0;
    for char in input.chars() {
        i += char as u32;
    }
    i
}

It runs an iteration in the for loop for the length of the string. This function has O(n)O(n)O(n) time complexity.

(Generally, look for loops to determine time complexity)

You always drop constants​

O(k⋅n)⟶O(n)O(k\cdot n) \longrightarrow O(n)O(k⋅n)⟶O(n)

While practically important, they're not theoretically important.

For example, if comparing O(10n)O(10n)O(10n) and O(n2)O(n^2)O(n2):

We're not concerned with the exact time, but instead, how it grows.

Practical and theoretical differences​

Just because O(n)O(n)O(n) is faster than O(n2)O(n^2)O(n2) doesn't mean it's always better.

Remember, we drop constants. Therefore, O(100n)O(100n)O(100n) is faster than O(n2)O(n^2)O(n2) but in the practical sense, it's slower for small inputs.

For example, in sorting algorithms, we use insertion sort for small inputs. Though slower in the theoretical sense because it's O(n2)O(n^2)O(n2), it's faster in the practical sense because it's faster for small inputs than for example, quicksort which is O(nlog⁡n)O(n\log n)O(nlogn).

Consider the worst-case scenario​

fn sum_char_codes(input: String) -> u32 {
    let mut i = 0;
    for char in input.chars() {
        if char == 'E' { // checks if char is 'E', then instantly breaks
            return i;
        }
        i += char as u32;
    }
    i
}

This function still has O(n)O(n)O(n) time complexity because we're looking for the worst-case scenario.

While there are some algorithms where it's better to reason using the best case and the average case, most of the time, we're concerned with the worst-case scenario.

In interviews, almost always assume the worst-case scenario.

Simple examples​

O(n2)O(n^2)O(n2)​

fn sum_char_codes(input: String) -> u32 {
    let mut i = 0;
    for char in input.chars() { // loop 1
        for char in input.chars() { // loop 2
            i += char as u32;
        }
    }
    i
}

For each character in the string, we iterate through the string again. This is O(n2)O(n^2)O(n2).

O(n3)O(n^3)O(n3)​

fn sum_char_codes(input: String) -> u32 {
    let mut i = 0;
    for char in input.chars() { // loop 1
        for char in input.chars() { // loop 2
            for char in input.chars() { // loop 3
                i += char as u32;
            }
        }
    }
    i
}

Just like the previous example, but we iterate through the string three times. This is O(n3)O(n^3)O(n3).

Other notations​

Ω\OmegaΩ​

Ω\OmegaΩ is the lower bound of an algorithm.

For example, Ω(n)\Omega(n)Ω(n) means that the algorithm will take at least nnn time.

Θ\ThetaΘ​

Θ\ThetaΘ is the tight bound of an algorithm.

For example, Θ(n)\Theta(n)Θ(n) means that the algorithm will take at least nnn time and at most nnn time.

Space complexity​

Space complexity is the amount of memory an algorithm takes up. Generally it's less consequential than time complexity.

Conclusion​

  • Big-O is a way to categorize algorithms based on time/memory.
  • Growth is with respect to input.
  • You always drop constants.
  • In most cases consider the worst-case scenario.
